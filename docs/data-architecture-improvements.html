DataSnoop · Data architecture · Strategy review

From a filing cabinet to a network

DataSnoop's database today is shaped by how data arrives — KBO files, NBB filings, gazette PDFs. To unlock PE-grade workflows and scale comfortably, the next evolution is to reshape it around how a user thinks: people, companies, ownership, events over time. This is a strategy review, not an implementation plan.

TL;DR — the whole plan in 5 lines

  • Today's schema is hub-and-spoke around the CBE. Great for "show me one company"; weak at "show me everyone connected, as of date Y".
  • Three big bets for interconnectedness: person as a first-class entity, ownership as a graph, time as a dimension.
  • Three search-performance non-negotiables for instant search: end-to-end async path, end-to-end query cancellation, full-text index on company names (trigram becomes the typo-fallback, not the primary).
  • Resilience & observability: verified backups + DR drills, structured logs + per-phase timing + alerts, explicit performance SLOs per surface.
  • Phased over six months. Two things ship in week 1 — the immediate async-route + threadsafe-pool bug fixes — because they explain today's "5-second waits" and the rest builds on a healthy foundation.

1. The metaphor: from filing cabinet to social network

If today's mental model doesn't fit, the schema will keep fighting back.

Today: a filing cabinet

DataSnoop's database is one giant filing cabinet. Each drawer is a company (the CBE is the drawer label). Inside each drawer, folders for KBO data, NBB financials, board members, gazette events. Pulling one drawer is fast. Pulling 5,000 drawers and asking "which people show up across them" is slow and clumsy.

Tomorrow: a social network

Reshape the same data as a network. Companies, people, addresses, events are nodes. Ownership, board seats, share transfers, residencies are edges with properties (start date, end date, percentage, role). Every node is a navigable starting point. Time is a dimension you can travel through.

The same three data sources still feed it. The same Postgres still stores it. What changes is the shape the application sees.

2. What's actually friction today

Five honest pain points. None are catastrophic; together they argue for the bigger reshape.

Friction 1 · Identity

People are strings, not entities

A "Jan Janssens" director at company A and a "Jan Janssens" director at company B are the same row in administrator only by coincidence of string equality. Two people with the same name in different cities → two rows. Same name in the same city → one row, merged accidentally. There's no person_id. Search groups by (name, domicile_city, domicile_postcode) which works well-enough but is heuristic, not authoritative. Every PE-relevant question — board interlocks, deal-team intel, "everyone Jan has been director of" — runs into this.

Friction 2 · Ownership

Subsidiary chains are flat rows, not a graph

shareholder says X owns Y%. participating_interest says X declares a stake in Y. To answer "what's the ultimate owner of this target" you write a recursive CTE by hand each time. Walking a holding tree five levels up isn't impossible — it's just always custom SQL. A single explicit ownership_edge table would make graph traversal trivial.

Friction 3 · Time

Every relationship reflects "the latest filing"

A director appointed in 2020, resigned 2023, reappointed 2024 lives as a row whose mandate_start / mandate_end tell you the current state. "Who was the board on 2022-06-01?" needs you to walk back through filing history manually. That's the kind of query a PE diligence customer expects to be one click. Today it's a custom investigation.

Friction 4 · Hot path

activity_log is on every API request, both ways

TierLimitMiddleware reads activity_log on every /api/* call to count today's usage. Every request also writes a row. The table has indexes, but it's a hot single table. At 10× scale it's the first thing that contends. Redis counters for the rate-limit fast path + async batch writes for the audit trail would absorb that without changing anything user-visible.

Friction 5 · Schema drift

Schema evolves by inline ALTER and runtime DDL

company_enrichment is created at runtime by a Python startup hook (_ensure_enrichment_table). Several columns get added by inline ALTER TABLE ADD COLUMN IF NOT EXISTS. src/schema.sql is run idempotently with CREATE IF NOT EXISTS everywhere. The result: nobody — including future-you — can tell what schema a fresh-install vs a two-year-old prod has, without reading the live database. A migrations folder with timestamped, replay-able files removes that ambiguity.

The unifying theme: the schema today reflects how data arrives (one source per cluster of tables, one filing per row, latest-state-only). The product wants entities, relationships, and history. Closing that gap is the next architectural job.

3. Three big bets

Each is large but bounded. Each unlocks an entire category of features. Together they are what "interconnected data" actually means.

Bet 1

Promote Person to a first-class entity

The single highest-leverage change. Today a "person" is the string you happen to read. Tomorrow it's a row in a person table, with a stable person_id that admin / shareholder / event rows reference.

What changes

A new person table. A new person_link table mapping every name occurrence in administrator, shareholder, and staatsblad_event to a person_id, with a confidence score. A nightly entity-resolution job (Splink, dedupe, or homegrown) that clusters rows using name + domicile + co-occurrence patterns. The product surfaces switch from "rows that happen to share a string" to "everything linked to person ID 4711".

What it unlocks

"Show me every company Jan has ever been a director of." Board interlock detection ("which boards have 3+ overlapping directors"). Deal-team intelligence ("this PE fund has installed person X as chair of every portfolio company"). A real Person profile page. Ranking signals like "people connected to many large companies" on search.

What it costs / risks

Engineering: 4–6 weeks for a v1. The risky bit is the entity-resolution model — false merges (two real people collapsed into one) are PR-bad. Mitigation: confidence-scored links + a conservative threshold + an "alternates" panel on the UI. KBO's direct-marketing prohibition still applies — the table is a derived data structure, not an export, and rate limits already gate the read paths.

Effort: M Unlock: very high Risk: false merges
Bet 2

Model ownership as a graph, not flat rows

Today, shareholder and participating_interest are two flat tables that reverse each other. Holding chains are implied; not modeled. Tomorrow, one ownership_edge table with an explicit graph structure, traversable in a single query.

What changes

Ingest both shareholder and participating_interest (and capital events from the gazette) into a single ownership_edge(parent_id, child_id, kind, pct, fiscal_year, source). Adopt Apache AGE — a Postgres extension that adds Cypher graph queries to the existing database. No new operational store; same backups, same monitoring.

A query that today is "five-level recursive CTE I write each time" becomes MATCH (target)-[:OWNED_BY*1..6]->(ultimate) RETURN ultimate.

What it unlocks

UBO trees. "What other companies does this PE fund own?" Network graphs on the profile page (the existing "spiderweb" view becomes a single graph query). Subsidiary roll-ups for valuation. Cross-group aggregate KPIs ("total revenue across this group's 40 entities").

What it costs / risks

Engineering: 2–3 weeks if AGE is adopted, 4–5 weeks for pure-SQL. Risk: AGE is a Postgres extension — adds an operational dependency, version upgrades need care. UBO data is genuinely incomplete (Belgium's UBO register isn't open-data); the graph will have known dead-ends. Be honest about that in the UI.

Effort: S–M Unlock: high Risk: AGE operational footprint
Bet 3

Make time a first-class dimension (bitemporal)

The most architecturally ambitious of the three. Every relationship becomes valid-from / valid-to plus a recorded-at timestamp. Snapshot queries become trivial. "What changed since last visit" becomes a one-liner.

What changes

On all relationship tables — administrator, shareholder, participating_interest, address, company_enrichment, even activity (NACE codes change too) — add valid_from, valid_to, recorded_at. Replace "the latest filing wins" with "every filing produces an immutable fact; current state is derived".

A query helper state_as_of(target_date) returns the consistent snapshot. The materialised tables (financial_latest etc.) remain — they're now simply "as of today" snapshots.

What it unlocks

"What did this company look like at the time of the previous deal?" Change-detection alerts ("a director at one of your favourites was replaced this week"). Historical diligence. Honest answers to "when did this group structure form".

What it costs / risks

Engineering: 6–8 weeks. Touches a lot of tables; needs a clean migration path with rollback. Storage grows (every fact lives forever) but Postgres compression handles this well. A real risk is query complexity creeping into application code — wrap it in views and a state_as_of helper to keep callers simple.

Effort: L Unlock: high Risk: migration scope

Why these three together

Person + ownership + time is the minimum interconnected model that lets you answer questions like "show me all the companies this person has ever sat on the board of, including subsidiaries of those, with revenue at the time they joined". That sentence is unanswerable with today's schema. After these three bets, it's a single graph query.

5. Search — performance first, ranking later

"Instant search" needs three non-negotiable performance properties before any relevance work matters. The ranking evolution stays on the roadmap but moves after performance.

5.1 Three non-negotiables for instant search

These three are the difference between "instant" and "5-second waits while the user types". Two of them are repo-grounded fixes (the async path and pool); one is a small new index pattern. Implementation specifics — schema, migration, downsides per item — live in the deep-dive doc's §5 Search performance.

Non-neg 1

Concurrent searches must not block each other

Today's pattern: search_companies is async def, but it calls _search_companies_cached which is sync and runs psycopg2 directly. Sync code inside an async route blocks the event loop thread. When the user types fast (j → je → jea → jean in <1 sec), four searches pile up and the last one waits for the first three to finish. That's where today's "5-second waits" come from.

What changes

Two-step fix: (1) wrap sync DB calls in asyncio.to_thread() so they run on a worker thread, not the event loop. Same-week win, 1–2 hours of work. (2) migrate to asyncpg for the search hot path so DB I/O is genuinely non-blocking and cancellable end-to-end. Bigger change but the right long-term shape.

What it costs / risks

(1) requires the SimpleConnectionPool → ThreadedConnectionPool swap or you immediately have a thread-safety bug. Trivial. (2) is more involved — touches the DB layer, may need separate sync/async pools, and either rewriting the most complex CTEs (search V2's seven-arm union) or keeping them on a sync path while everything else goes async.

Effort: S → M Unlock: solves head-of-line blocking Risk: requires pool swap or thread-safety bug
Non-neg 2

Cancellation must propagate end-to-end

The browser already cancels stale requests via AbortController. But that only closes the network socket — the database keeps grinding on the cancelled query. While the user has moved on to keystroke 7, the server is still finishing keystrokes 3 and 4, holding connections, blocking later requests.

What changes

With asyncpg (Non-neg 1's path 2), task cancellation propagates to the underlying connection automatically — the right pattern for free. Bridge while still on psycopg2: a watchdog task per request that polls Request.is_disconnected() and, on disconnect, calls connection.cancel() from a separate connection. Both patterns are documented; pick one based on when you adopt asyncpg.

What it costs / risks

The watchdog bridge is ~40 lines of middleware code. asyncpg handles it natively. The risk is a cancelled query leaving half-applied state — but search is read-only, so cancellation on a SELECT is always safe.

Effort: S Unlock: capacity reclaimed on every keystroke Risk: low; only applies safely to reads
Non-neg 3

Full-text index on company names, not pattern matching

Common words — consulting, holding, jean — trigger a fuzzy trigram scan over ~570k embedded vectors and ~1.94M company-name rows. Cold-cache latency on these terms is ~1 second. The proper solution is a precomputed tsvector + GIN full-text index. Trigram matching stays as the typo-fallback, not the primary path.

Already exists in the codebase — backend/routers/staatsblad_events.py:295 uses to_tsvector('simple', ...) + plainto_tsquery on event text. Pattern is proven; it just needs to land on the name-search path too.

What changes

Add a name_tsv generated column to company_info as to_tsvector('simple', name_normalized), and a GIN index on it. Rewrite the first arm of _SEARCH_SQL to use websearch_to_tsquery. Trigram match becomes the fallback when FTS returns <5 hits or query is <4 chars.

What it costs / risks

One ALTER TABLE + one CREATE INDEX, both online with CREATE INDEX CONCURRENTLY. Re-runs the _SEARCH_SQL CTE structure — needs A/B testing against the existing scored arms to verify quality didn't regress on edge cases (legal-suffix synonym handling, accent stripping). Bench against a sample of real queries from activity_log.

Effort: S Unlock: ~10× faster on common-word queries Risk: regressions on edge cases — bench first
The non-negotiable ordering: Non-neg 1 + 3 ship first because they have direct user-visible latency impact. Non-neg 2 ships alongside or just after. The ranking work below (5.2) is the polish that follows performance.

Why not yet — and the new precondition

Two reasons to defer ranking work:

  • Volume. The ML re-ranker pays off when (a) you have enough click data to learn from, and (b) the cost of bad relevance is high enough to justify the infrastructure. Today's volume — small.
  • The 5.1 non-negotiables come first. A learned re-ranker on top of an event-loop-blocked, uncancellable, trigram-scanning search adds a CPU layer to a slow path. Fix the path first; rank afterwards.

Realistic timeline: Q3 or later. The performance non-negotiables in 5.1 ship in week 1–4; the ranking work follows once they're stable.

What features I'd train on, in priority order
  1. Query-document text similarity — already implicit in scored arms; explicit feature for the model.
  2. Click-through history — from activity_log joined to a click event.
  3. Financial signal — log(revenue), is-active, has-recent-filing.
  4. Recency — most recent NBB filing date, most recent gazette event.
  5. Popularity — already computed in company_popularity.
  6. Sector match — does the query mention NACE-relevant tokens?
  7. Person/company hybrid — if query is person-like, weigh person results.
  8. Embedding cosine — the existing semantic feature, as a column.

6. Resilience & observability

"Bullet-proof" means: when things break, you know first, you have a tested recovery path, and the user impact is bounded. Today the system is single-Postgres, single-VPS, with no proven backup-restore drill and no per-request observability.

6.1 Backups + verified restore drills

Foundation in place (2026-04-30): dedicated 100 GB ext4 Hetzner volume #105558877 mounted at /mnt/volume-hel1-1. Subdirectories ready: $DS_WAL_ARCHIVE_DIR for continuous WAL archiving and $DS_BACKUP_DIR for base-backup landing. The path-discovery problem (where do backups go?) is solved; the work below is the script + cron to actually populate them.
r10 caveat — WAL pressure during restore: the volume holds archived WAL, not live WAL. Postgres writes $PGDATA/pg_wal/ to the root disk regardless of where archive_command ships rotated WALs. A large pg_restore into the same Postgres instance can balloon live WAL transiently (a 27 GB clone can generate >5 GB of pg_wal) even when archiving is configured. Preflight before any restore: verify root disk has at least 15 GB free, OR run the staging Postgres as a separate process whose $PGDATA lives on the volume — that's the only way to keep restore WAL off root entirely. The Phase-1 plan accepts the same-instance approach with the preflight; the separate-process pattern moves in for production-grade resilience.

Today's state: Postgres on a single Hetzner VPS with local-disk storage. No standby replica. No nightly verified-restore drill — meaning we don't know if a restore would work until we need it. Volume is in place; archiving config + restore drill cron are not yet built.

What changes:

  • WAL archiving to S3-compatible object storage (Hetzner Object Storage, Backblaze B2). Continuous WAL ship + nightly base backup gives Point-in-Time-Recovery. RPO target: 5 minutes.
  • Monthly automated restore drill. Spin up a throwaway Postgres on an EC2-style instance, restore the latest base + WAL up to T-1h, run a 10-query smoke test, tear down. Email summary to operator. Drill failure = page.
  • RTO target: 1 hour for full recovery from cold backup. Realistic at our DB size (~27 GB).
  • Standby replica (Phase 2). Streaming replication to a second small VPS in a different Hetzner zone. Promotes to primary in <5 min if main VPS dies. Doubles infra cost slightly but adds an actual recovery path.

6.2 Observability — structured logs, metrics, traces

Today's state: stdout logs go to Docker; no structured fields, no aggregation, no alerts on error rate. The per-request phase timing middleware in §4 is the foundation; below is what to build on top.

LayerWhat shipsWhy
Structured logs Switch logger calls to JSON output (structlog or stdlib + python-json-logger). Required fields: request_id, user_email (or anon:hash), endpoint, status, duration_ms, error_class. Greppable by user / endpoint / error / time-range. Foundation for everything else.
Metrics Prometheus client lib + /metrics endpoint behind admin auth. Histograms for per-endpoint latency, counters for error class, gauges for queue depth + connection pool usage. Aggregation across requests. Drives the SLO dashboards.
Distributed traces OpenTelemetry SDK in the backend, traces shipped to a managed collector (Honeycomb free tier, or self-hosted Tempo). One span per phase from the §4 timing middleware. "Why was THIS request slow?" — answer in one trace, not 30 minutes of custom telemetry. Solves the original observability gap directly.
Alerts Page on: error rate > 1% over 5 min, p95 latency > SLO over 15 min, backup-restore drill failure, queue worker stuck > 30 min, NBB key rotation needed. Route to operator email + (eventually) on-call rotation. Find out from the system before the customer.

6.4 Schema migration safety

  • Migrations runner ships first (already in §4 same-week wins). After that, every migration is replayable, ordered, and dependency-tracked.
  • Pre-deploy migration dry-run on staging. Once staging DB is real (§4 structural), the deploy script applies migrations to staging, smoke-tests, then applies to prod. Fail-fast on staging migration failure.
  • Online-DDL patterns enforced. No ALTER TABLE with default-value backfill on multi-million-row tables during business hours; batch backfills via a Python script with throttling. Bitemporal Phase 0 is the load-bearing case.
  • Reversible by default. Every forward migration ships with a paired rollback migration (current convention in migrations/2026-04-2X_*_rollback.sql — keep this).

7. Radical options — weighed

Things that get suggested at this scale of company. My honest verdict on each.

OptionVerdictReason
Apache AGE (graph queries on Postgres) Yes Graph traversal without a second operational store. Same backups, same monitoring. Direct enabler for Bet 2.
TimescaleDB extension for financials Maybe Financials are time-series — TimescaleDB does FY rollups and continuous aggregates well. Worth a 2-day spike. Not strictly needed.
Bitemporal modelling (Bet 3) Yes Plain-Postgres, no extension. Schema work is real but pays off across many features.
Standalone graph DB (Neo4j, Memgraph) No Two operational stores, two backup regimes, two query languages. Postgres + AGE gets 80% for 20% effort.
Move analytics to ClickHouse / DuckDB No Premature. Postgres handles 200M rows fine with the right indexes. Revisit at 10× volume.
Vector DB (Qdrant, Pinecone, Weaviate) No pgvector with HNSW handles tens of millions. The ops cost of a vector DB is not justified at our 600k-vector scale.
Microservices split (separate enrichment service, etc.) No Solo founder. One DB, three containers. Stay there until you have engineering team scaling to motivate it.
Event sourcing (everything is an event log) No Conceptually appealing; operationally heavy. Bitemporal tables get you 90% of the value for 10% of the cost.
CQRS / read models Yes — already doing it financial_latest, company_info, sector_percentiles ARE read models. Formalise the pattern; new read models become routine instead of bespoke.
Kafka / event-driven backbone No Postgres LISTEN/NOTIFY + cron + worker queues handles every async pattern at this scale.
OLAP cube / BI layer (Cube.js, Metabase, etc.) Maybe For internal admin analytics, Metabase on a read replica is cheap and useful. For customer-facing it's not the bottleneck.

10. What NOT to do

The shortlist of seductive ideas that would make things worse at this stage.

Avoid these unless something specific forces them

  • Don't introduce a separate graph database. Postgres + Apache AGE covers it. Two operational stores doubles the ops surface for one product.
  • Don't move to microservices. One Postgres, three containers, one deploy script — keep it. Microservices solve team-scaling problems you don't have.
  • Don't move analytics to ClickHouse / a data warehouse yet. Postgres handles the volume. Revisit after Person + Ownership ship and you've measured a real bottleneck.
  • Don't try to model UBO from KBO data alone. The data isn't there. Wait for the Belgian UBO register API or partner with Graydon / Companyweb. Otherwise you're inferring with no good signal.
  • Don't normalise activity_log into something fancy. Partition by month, archive cold months, fast-path with Redis. That's the whole answer.
  • Don't refactor the AI pipelines into a single one. Bulk + narrative are intentionally different beasts (cost vs. quality). The current split is correct.
  • Don't bypass tier limits even temporarily. They are load-bearing. The TierLimitMiddleware fast-path fix is the right answer; turning it off "for now" is technical-debt poker you will lose.
Strategy review for internal discussion. Not an implementation plan and not a promise — it's the sequence I'd argue for if I were owning the DataSnoop codebase for the next two quarters. Numbers and scopes are directional; before any phase ships, scope it concretely against the live DB and the customer pipeline.
