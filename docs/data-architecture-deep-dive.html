Execution model — Codex implements, Claude reviews (r19, operator decision 2026-04-30). The work proceeds in a hybrid mode. Each stage / step has TWO actors with non-overlapping responsibilities.
ActorRoleWhy this side wins
Codex (cloud) Implements the stage's actions: file edits, schema migrations, deploys, tests. Operates in an isolated sandbox per stage. Parallel cloud sandboxes; clean execution environment per task; different model perspective on the implementation pass than the model that wrote the plan.
Claude Code (local) Reviews the staged work before merge. Verifies postconditions. Spawns the two review agents (correctness + security) per the standing rule. Coordinates operator approvals. Operator's MEMORY.md profile, CLAUDE.md standing rules, existing skills (security-review, correctness-review), and familiarity with the doc and conventions all live in .claude/. Claude reads what Codex produced and applies the project's review discipline.

Per-stage handoff loop

  1. Operator hands Codex the stage's block (preconditions / actions / files / postconditions / approval-gate, per the per-step format below).
  2. Codex verifies preconditions in its sandbox, executes the actions, emits the postcondition output (queries, curl, grep, log excerpts). Pushes the work to a feature branch off master.
  3. Operator hands Claude the PR + the postcondition output + a pointer to the stage in this doc.
  4. Claude reads the diff, verifies postconditions independently, spawns the two review agents (correctness + security) per CLAUDE.md's standing rule, and reports green/red with specifics.
  5. If green: operator approves the merge. The standing rule "production deploy requires explicit operator approval after staging is green" still applies — Claude review ≠ prod deploy approval.
  6. If red: Claude flags issues with file:line references; operator decides whether Codex retries (with the issues spelled out) or the stage is paused for re-spec.

What this changes in the doc

  • Each step's Postconditions section is now the formal contract between Codex (emits) and Claude (verifies). Vague postconditions = unclear handoff. Aim for queries/greps/HTTP responses, not "should look right".
  • The standing two-review-agent rule ("correctness + security before merge to master") is still required — Claude is the side that runs them. Codex does NOT self-review.
  • Operator approval still gates prod deploy. Claude review passing means the diff is mergeable; deploy is a separate gate the operator decides.
  • Branch convention: Codex works on a feature branch named for the stage (e.g. feat/migrations-runner, feat/person-v1), not directly on master.

Revision 25 (2026-04-30): five P-rated correctness items from the next operator review pass + one sequencing fix to the Week 1 bundle. One P1 (replay-gate baseline rule was ambiguous); four P2 (current views drop unknown-date facts; person_link assumed one mention per source row; external/unknown ownership IDs lacked a stable construction convention; cancellation watchdog's pool-return path could poison the next request). The Week 1 split is the operator's flagged-as- riskiest item: five coupled deliverables in one slot was the doc's worst scheduling assumption. Convergence is still strong — no SQL bugs that fail at execution this round, but several real data-model gaps that would have surfaced as silent corruption.
  1. P1 — Fresh-install replay needed one canonical baseline rule. The r22 spec said "apply src/schema.sql baseline → apply every migration in order". Ambiguous: if src/schema.sql already includes objects introduced by historical migrations, replaying both produces duplicate-create errors (or worse, IF NOT EXISTS guards mask drift silently). Picked Rule A: src/schema.sql is THE canonical baseline; migrations/ holds POST-baseline forward deltas only. Schema file carries an explicit -- BASELINE_AS_OF: <date> header; runner only applies migrations dated after it. Pre-cutover task: fold the 4 existing forward migrations into src/schema.sql, bump the header to BASELINE_AS_OF: 2026-04-28, archive the 8 files into migrations/_archived_<date>/. Rule B (migrations-only, schema.sql becomes a generated reference dump) considered and rejected — extra work that buys nothing v1 doesn't get from Rule A.
  2. P2 — _current views dropped unknown-date facts. The r23 fix added valid_from <= CURRENT_DATE, which is correct for fully-backfilled rows but lossy when backfills have gaps (legacy KBO rows from before NBB structured deposit_date; gazette events without a structured start). r25 default: accept (valid_from IS NULL OR valid_from <= CURRENT_DATE) with NULL documented as "unknown start; treated as in-force at least since deposit". Per-table, post-backfill, can tighten to NOT NULL once data quality is verified (Stage postcondition: SELECT count(*) FROM <t> WHERE valid_from IS NULL = 0). Applied to administrator_current and ownership_edge_current; same convention documented for the equivalent shareholder / PI / affiliation views when they're sketched.
  3. P2 — person_link assumed one mention per source row. UNIQUE (source_table, source_pk) forces extractors to invent artificial composite source keys when one source row mentions multiple persons (e.g. staatsblad event "Jan and Marie were appointed directors"; share_transfer with both transferor and transferee names). Added source_mention_seq INTEGER NOT NULL DEFAULT 0 to the row + the UNIQUE; optional source_field for extractors that can name the field. Same pattern ownership_edge already uses with source_action_seq.
  4. P2 — Unknown/external parent_id needed a stable construction convention. r21 added external_org + unknown kinds to ownership_edge but didn't specify how their parent_id values are generated. Without a convention, the same external owner (e.g. a Dutch parent NV) gets multiple parent_id values across rows → graph fragments; or two different unknown name-only owners collide → graph wrongly merges. Added the convention as a write-time rule: external_org uses scheme:value (e.g. LEI:5493001KJTIIGC8Y1R12, NL_KvK:34123456); unknown uses unknown:<sha256-hex-prefix-16>(search_normalize(name) || '|' || country). Helper at backend/ownership_id.py; CHECK constraint enforces format per kind so loader bypasses fail at write.
  5. P2 — Cancellation watchdog poisoned the connection pool. The r20/r21 watchdog correctly guards pg_cancel_backend() with PID + application_name, but the spec didn't say what to do with the psycopg2 connection AFTER QueryCanceled fires. The connection sits in aborted-transaction state; returning it straight to the pool means the next caller's first statement errors with InFailedSqlTransaction on a query they didn't run. Added: handler's exception path does conn.rollback() for the search read path (cheap); pool.discard(conn) for any future heavier path that opens server-side cursors / temp tables / session state. Pool-side safety net: db.py::put_connection() checks conn.info.transaction_status and discards anything not TRANSACTION_STATUS_IDLE regardless of caller intent. Native asyncpg path is unaffected (task cancellation handles connection state automatically).
  6. Sequencing — Week 1 split into 1a / 1b / 1c / 1d (operator-flagged as the doc's riskiest scheduling assumption). Previous Week 1 rolled up: migrations runner + runtime-DDL inventory + replay CI gate + style contract + phase-timing middleware. Five coupled deliverables in one slot meant any one slipping stalls every later week. Split: 1a = minimal runner (up/status/baseline; tx/no-tx headers; advisory lock; nothing else); 1b = runtime-DDL inventory + capture (depends on 1a); 1c = CI gates (syntactic, replay, style) + dry-run + checksums (depends on 1b — running gates against the known-red existing 14 sites is bad ergonomics); 1d = phase-timing middleware (parallel, no migration-stack dependency, can ship by a different reviewer). Deploy-hook staged-rollout (per r23) anchors to 1a's completion specifically, not "Week 1 generally".

Each of the four proposals (Person / Ownership / Bitemporal / Staging) and the search-perf section adds new query paths or new data tables. Every one of them inherits the same set of performance and operational requirements. A proposal isn't "ready" unless it satisfies all of these. -- New: canonical Person record. v1 deliberately minimal.
-- ⚠️ r16: merged_into + status added — the GDPR mitigation prose
-- in §1's "Hard prerequisites" callout references both columns; they
-- need to actually exist for the redirect / erasure logic to compile.
CREATE TABLE person (
    person_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    canonical_name      TEXT NOT NULL,
    name_normalized     TEXT GENERATED ALWAYS AS (search_normalize(canonical_name)) STORED,
    primary_city        TEXT,
    primary_postcode    TEXT,
    role_count          INT DEFAULT 0,                -- denormalised for ranking
    first_seen_date     DATE,
    last_seen_date      DATE,
    cluster_version     TEXT,                          -- which resolver run produced this row
    status              TEXT NOT NULL DEFAULT 'active'
                        CHECK (status IN ('active', 'merged', 'tombstone')),
    merged_into         UUID,                          -- if status='merged', points at the surviving person_id
    created_at          TIMESTAMPTZ DEFAULT NOW()
    -- NOTE: estimated_birth_year intentionally NOT in v1. Adding a guessed
    -- field signals a confidence we don't actually have. Revisit when we
    -- have a way to anchor it (e.g. mandate-start of first young role).
);

-- Backend handler at /person/<id> reads `status` first:
--   active    → 200 OK, render profile
--   merged    → 301 Moved Permanently → /person/<merged_into>
--   tombstone → 410 Gone (GDPR erasure executed; no redirect)

-- New: link every name occurrence to a Person, with confidence.
-- ⚠️ r25 — UNIQUE key includes source_mention_seq because one source row
-- can mention multiple persons. Examples:
--   - staatsblad_event "Jan and Marie were appointed directors" → 2 mentions
--   - share_transfer event with both transferor and transferee names → 2 mentions
--   - future extractor improvements that pull more names per row
-- The original UNIQUE (source_table, source_pk) assumed exactly one
-- mention per source row, which forces extractors to invent artificial
-- composite source keys when reality has multiple mentions. Adding
-- source_mention_seq (0-indexed, default 0 for back-compat with
-- single-mention extractors) is the same pattern ownership_edge already
-- uses with source_action_seq (see §2 schema sketch).
-- Optional source_field for extractors that can name the field
-- (e.g. 'transferor_name', 'transferee_name') — informational only,
-- not part of the UNIQUE because seq alone enforces row-level uniqueness.
CREATE TABLE person_link (
    id                  BIGSERIAL PRIMARY KEY,
    person_id           UUID NOT NULL REFERENCES person(person_id),
    source_table        TEXT NOT NULL,                 -- 'administrator' | 'shareholder' | 'staatsblad_event'
    source_pk           TEXT NOT NULL,                 -- composite key serialised
    source_mention_seq  INTEGER NOT NULL DEFAULT 0,    -- r25: 0..N mentions per source row
    source_field        TEXT,                          -- r25: optional named field e.g. 'transferor_name'
    enterprise_number   TEXT,
    name_as_written     TEXT,
    link_kind           TEXT NOT NULL,                 -- 'deterministic' | 'probabilistic' | 'manual'
    confidence          REAL NOT NULL,                 -- 0..1, deterministic = 1.0
    confirmed_by_human  BOOLEAN DEFAULT FALSE,
    cluster_version     TEXT,                          -- which resolver run produced this link
    UNIQUE (source_table, source_pk, source_mention_seq)
);

-- New: full audit history for merges, splits, and manual corrections.
-- Replaces the simpler "merged_into" pointer with a real log so we can
-- always reconstruct how a cluster got to its current shape.
CREATE TABLE person_merge_log (
    id              BIGSERIAL PRIMARY KEY,
    op_kind         TEXT NOT NULL,                  -- 'merge' | 'split' | 'manual_correct' | 'tombstone'
    primary_id      UUID NOT NULL,                  -- the surviving / canonical person_id
    secondary_id    UUID,                              -- the merged-from / split-from id (NULL for tombstone)
    moved_link_ids  BIGINT[],                          -- person_link.id values that switched cluster
    op_at           TIMESTAMPTZ DEFAULT NOW(),
    op_by           TEXT,                              -- 'resolver_v3' | admin email for manual ops
    reason          TEXT
);

CREATE INDEX idx_person_link_person ON person_link(person_id);
CREATE INDEX idx_person_link_enterprise ON person_link(enterprise_number);
CREATE INDEX idx_person_link_kind ON person_link(link_kind);
CREATE INDEX idx_pml_primary ON person_merge_log(primary_id);

-- Cross-cutting requirement: every new search surface ships FTS-first +
-- trigram-fallback. /person/<id> is a new search surface; mirror §5c.
-- Expression GIN indexes — no STORED column added.
CREATE INDEX CONCURRENTLY idx_person_name_tsv
  ON person USING GIN (to_tsvector('simple', name_normalized));
CREATE INDEX CONCURRENTLY idx_person_name_norm_trgm
  ON person USING GIN (name_normalized gin_trgm_ops);
-- Application path: FTS first via websearch_to_tsquery(search_normalize(:q));
-- trigram fallback only if FTS underflow (< 5 hits) AND len(query) ≥ 4.
-- Same leading-suffix asymmetry guard as §5c applies (NV Colruyt-style
-- queries need application-layer leading-suffix-stripping before FTS).

The original administrator, shareholder, and staatsblad_event tables stay completely unchanged. person_link is the bridge — every row in those tables that mentions a person gets one row here, with a link_kind that distinguishes deterministic vs. probabilistic vs. manual links.

How it gets built — v1 is deterministic-only

The original draft jumped straight to a Splink probabilistic resolver. Tightened path: ship a small, high-precision v1 first; earn the right to a fuzzy resolver in v2.

  1. Migration 1: create the three tables empty. No data movement, no resolver, no UI yet.
  2. Tiered deterministic linker — postcode availability is uneven across source tables, so v1 splits into three tiers:
    • Tier A — anchor on staatsblad_event.person_domicile_*: this is the only structured-postcode source (added by the prompt_v3 re-extractor; progressively backfilled). Cluster on (name_normalized, person_domicile_postcode, person_domicile_city). These become the high-precision seed clusters with confidence=1.0. Catalog-drift prerequisite: the person_domicile_* columns live in code (backend/routers/people.py:336–359) but are NOT in src/schema.sql. A baseline migration that captures them as canonical schema must ship before this work — otherwise a fresh-install Postgres will not have the columns the resolver depends on.
    • Tier B — fold in administrator, shareholder, AND affiliation rows by co-occurrence: a Tier-B row matches a Tier-A person if name normalises to the same string AND they appear together on at least one common enterprise_number across filings. confidence ≈ 0.9; link_kind='deterministic' still acceptable because the co-occurrence anchor is hard. The affiliation table (defined in migrations/2026-04-25_affiliation.sql, capturing natural-person representatives behind legal-person admins) was missing from the r3 sources list — added in r5. Affiliation rows are particularly useful for resolving the "Jan Janssens represents NV X which is a director of NV Y" pattern.
    • Tier C — residual rows: rows with no Tier-A match (admin rows for people who never appear in a structured staatsblad domicile, etc.) get a singleton person_id per row with confidence=0.5. These are the candidates the v2 probabilistic resolver will work over.
    Belgian-postcode regex extraction from shareholder.address / participating_interest.address free-text fields can lift some Tier-C rows into Tier B; deferred to a v1.1 if precision proves OK.
  3. Golden review set — stratified, ~200 v1 / ~500 before public auto-merge (r17). Hand-build a labelled set of known person identities sourced from operator interviews + Belgian PE reference data. Stratification matters more than raw count — a 200-row set covering only common Belgian surnames will pass tests but fail in production. Required strata:
    • Common Belgian surnames (Janssens, De Smet, Peeters, Lambert) — high homonym density
    • Multilingual / accent variants of the same person (Pieter / Peter; François / Francois; Müller / Muller)
    • Foreign directors with non-Belgian domicile (German, Dutch, French) — Tier-A's person_domicile_* match doesn't help here
    • Legal-representative chains (natural person represents legal-person admin — the affiliation table case)
    • Same-city homonyms (two real "Jan Janssens" both in Brussels with no domicile postcode disambiguation) — these are the false-merge traps the golden set MUST cover
    • Known false-merge regression cases from past customer reports / operator-flagged mistakes
    200 examples is fine for v1 smoke testing of the deterministic linker. ~500 examples spanning all six strata is the minimum to trust an auto-merge that's exposed via a public URL. Use it to measure precision/recall of any future probabilistic resolver. Without this set, "the resolver is good" is a feeling, not a measurement.
  4. Frontend (read-only first): a /person/<person_id> page that lists role history. No URL exposed publicly until the legal opinion is in. Per the cross-cutting requirements: this page is a new search surface — it must ship with FTS on person.name_normalized (not just trigram), async route handler, cancellation middleware, and per-phase timing. Don't re-create the company-search performance issue on the people path.
  5. Probabilistic resolver — v2 only: Splink (or alternative) on the unmatched residual. Hand-tune blocking + comparisons against the golden set. Threshold policy stays as before (≥0.95 auto, 0.7–0.95 review queue), but is calibrated to measured precision, not assumed.
  6. Nightly resolver: incremental on new data. person_id values are never reassigned; cluster changes go through person_merge_log (merge / split / manual_correct / tombstone) so the history is always reconstructable.

Resolver idempotency rules (r15)

The resolver runs nightly. Run N+1 sees Run N's output PLUS new source rows that landed since. Without explicit idempotency rules, person_ids could churn nightly and break stable /person/<id> URLs. These four rules govern the run-N+1 behaviour:

  1. Determinism on same input. Tier-A/B/C thresholds are configuration constants, not random. Re-running the resolver on identical input produces identical output. Anything resembling random tie-break gets a stable seed.
  2. Existing person_link rows are sacred. Run N+1 reads existing person_link first; the (source_table, source_pk) → person_id assignments from prior runs are taken as ground truth. The resolver only proposes changes to those assignments, never reassigns silently.
  3. New rows: assign or mint. For source rows that didn't exist in run N, the resolver runs Tier A → B → C as normal. Tier-A or Tier-B match → assign existing person_id. No match → mint new person_id (UUID gen_random_uuid). New person_link row inserted; no merge_log entry needed (it's not a merge, it's a fresh link).
  4. Cluster-change proposals go through person_merge_log, never silent overwrites.
    • Merge proposed (resolver thinks two existing person_ids should be one): write a person_merge_log row with op_kind='merge', primary_id=surviving, secondary_id=merged-from; UPDATE matching person_link rows to point at primary_id; the merged-from person row stays in the table with merged_into set so old URLs redirect properly.
    • Split proposed (resolver thinks one existing person_id should be two): mint a NEW person_id for the split-off cluster, write a person_merge_log row with op_kind='split', move the affected person_link rows to the new id. Old id keeps the majority of its links; URL stability preserved for the majority.
    • Never: reassign an existing person_id to mean "different person now". Always mint new for splits; always log for merges.

High

KBO data is public-registry; we already enforce no-marketing via tier limits. But aggregating one person's mandates, addresses, and gazette mentions into a single profile feels materially different from "scattered rows in source tables", even though the underlying data is the same. Could attract regulatory or customer-pushback we don't see today.

Mitigation: brief a Belgian privacy lawyer before shipping the profile page; document the legal basis (public-registry + journalism/research interest); make "right to erasure" handling explicit in the design (write a person_merge_log row with op_kind='tombstone' referencing the person_id; backend returns 410 from /person/<id> when a tombstone is the latest log entry); never expose the table via bulk export.

Medium

False splits (one real person scattered across IDs)

The opposite failure: "Jan Janssens" with one filing using "J. Janssens" and another using "Jean Janssens" stays as two person_ids. Less visible to users, but reduces the headline value ("show me everything Jan has ever done" returns 4 of 7 mandates).

Mitigation: the human-review queue catches the medium-confidence band; iterate on the resolver's comparison features (initials, accent variants, hyphenated names).

Medium

Stable URL contract is real engineering

/person/UUID-abc needs to keep working forever. If the resolver re-runs and re-clusters, person_ids cannot be reassigned without breaking external links and bookmarks. The merged_into pattern handles merges; splits are harder (which of the two new IDs gets the old URL?).

Mitigation: never reassign IDs; for splits, keep the old ID pointing at one of the new clusters and create a new ID for the other; handle redirect cases in the backend.

Medium

Splink on ~1.15M administrator rows + ~43K shareholder rows + ~staatsblad-event person rows is real work. First full run probably 30–60 min on the current 4-vCPU VPS. Incremental runs much cheaper. Needs sizing headroom alongside the existing nightly cron load.

Mitigation: incremental runs after the first; run during the existing 02:00–04:00 quiet window; if it becomes a problem, move to a beefier worker container or a cheap separate VPS.

Medium

Confidence scores are confusing for non-technical users

Showing "this is 87% likely the same Jan Janssens" is honest but most users will either ignore it or distrust the whole feature. PE analysts care; casual users don't.

Mitigation: show a binary "verified / probable" badge by default with the score available on hover; only show numbers in advanced views.

Foreign-name resolution will be weaker

"Hans Müller" of "Munich" with no Belgian domicile data — the resolver tuned for Belgian patterns will under-perform. Most foreign directors are edge-case for our PE-Belgian customer, so low priority.

Mitigation: accept lower confidence for foreign-domicile records; potentially exclude from the auto-merge band entirely.

Mitigation: accept that the medium band is "unprocessed" and queries fall back to the safe behaviour (separate person_ids); revisit with crowdsourced feedback (Pro tier "merge / split" buttons that earn the user karma).

-- v1 is pure SQL only. AGE is deferred until SQL CTEs and the existing
-- Python BFS in network.py prove insufficient.
--
-- ⚠️ r16 corrections to this schema:
--   - source_action_seq column added (one staatsblad event can produce
--     multiple actions: share_transfer = 2, ma_event:merger = N+1)
--   - UNIQUE constraint includes source_action_seq
--   - CHECK constraint on pct (0..100 or NULL)
--   - canonical view exposes parent_kind / child_kind

CREATE TABLE ownership_edge (
    id                  BIGSERIAL PRIMARY KEY,
    -- Endpoints + direction (parent OWNS child)
    -- ⚠️ r21: real-world ownership data includes foreign companies, funds,
    -- trusts, unstructured names with no identifier. r20's company|person-only
    -- model rejected those. Schema now accepts external_org + unknown kinds
    -- with explicit identifier-scheme + country fields.
    --
    -- ⚠️ r25 — parent_id CONSTRUCTION CONVENTION (load-bearing — without it,
    -- duplicate external owners fragment on import or accidentally merge on
    -- read). Generated at WRITE time by the loader, never composed at query
    -- time. Per parent_kind:
    --   'company'       → bare 10-digit CBE (existing convention; matches src/schema.sql)
    --   'person'        → person_id UUID (existing convention)
    --   'external_org'  → "<scheme>:<value>" — e.g. "LEI:5493001KJTIIGC8Y1R12",
    --                     "NL_KvK:34123456", "FR_SIREN:552120222". Same scheme
    --                     across rows produces same parent_id → no fragmentation.
    --   'unknown'       → "unknown:" || encode(sha256(
    --                          search_normalize(parent_name_raw) || '|' ||
    --                          COALESCE(parent_country, '')), 'hex')[1..16]
    --                     — name-only owners get a stable hash so the same
    --                     unstructured name+country combo always resolves to
    --                     the same parent_id. 16-hex-char prefix is collision-
    --                     safe at our scale (~1M rows ≪ 2^32).
    -- Helper function lives at backend/ownership_id.py — every loader writing
    -- to ownership_edge calls it; never construct parent_id by hand.
    -- CHECK constraint below (added in same migration) enforces format per kind.
    parent_kind         TEXT NOT NULL CHECK (parent_kind IN (
                                          'company',        -- Belgian CBE-identified
                                          'person',         -- resolved to person_id (UUID)
                                          'external_org',   -- non-Belgian co, fund, trust, etc.
                                          'unknown'          -- name only, no identifier
                                        )),
    parent_id           TEXT NOT NULL,
    parent_name_raw     TEXT,                           -- r21: unstructured name from filing/gazette
    parent_identifier_scheme TEXT,                      -- 'CBE' | 'LEI' | 'NL_KvK' | 'FR_SIREN' | 'UUID' | NULL
    parent_identifier_value  TEXT,                      -- the identifier under that scheme
    parent_country      CHAR(2),                        -- ISO-3166 alpha-2; NULL = unknown
    child_kind          TEXT NOT NULL CHECK (child_kind = 'company'),
    child_id            TEXT NOT NULL,                 -- always a Belgian CBE — child is always the subject

    -- r21: scheme-aware ID validation. CBE = 10 digits; person = UUID;
    -- external_org = identifier_scheme + identifier_value carry the validation;
    -- unknown = name_raw required, no identifier.
    -- r23 — use IS NOT DISTINCT FROM for the scheme equality checks. Plain `=`
    -- against a nullable column evaluates to NULL when the column is NULL, and
    -- Postgres treats CHECK predicates that evaluate to NULL as PASSING. So a
    -- 'company'-kind row with parent_identifier_scheme = NULL would slip through
    -- the CHECK silently. IS NOT DISTINCT FROM 'CBE' returns FALSE on NULL,
    -- which makes that branch FALSE, the OR FALSE, and the row correctly rejected.
    -- r25 — external_org and unknown branches now enforce the parent_id
    -- construction convention documented above (matches the helper at
    -- backend/ownership_id.py — schema-level guard catches loader bypasses).
    CHECK (
      (parent_kind = 'company'      AND parent_id ~ '^[0-9]{10}$'
                                      AND parent_identifier_scheme IS NOT DISTINCT FROM 'CBE')
      OR (parent_kind = 'person'    AND parent_id ~* '^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$'
                                      AND parent_identifier_scheme IS NOT DISTINCT FROM 'UUID')
      OR (parent_kind = 'external_org' AND parent_identifier_scheme IS NOT NULL
                                          AND parent_identifier_value IS NOT NULL
                                          AND parent_id = parent_identifier_scheme || ':' || parent_identifier_value)
      OR (parent_kind = 'unknown'   AND parent_name_raw IS NOT NULL
                                      AND parent_identifier_scheme IS NULL
                                      AND parent_id ~ '^unknown:[0-9a-f]{16}$')
    ),
    CHECK (child_id ~ '^[0-9]{10}$'),

    -- Quantity
    pct                 NUMERIC(5,2)
                        CHECK (pct IS NULL OR pct BETWEEN 0 AND 100),  -- r16: validate range
    edge_kind           TEXT NOT NULL,                 -- 'shareholder' | 'participating' | 'gazette_capital' | 'gazette_transfer' | 'gazette_ownership' | 'gazette_ma'

    -- Source provenance — every edge knows exactly where it came from
    source_table        TEXT NOT NULL,                 -- 'shareholder' | 'participating_interest' | 'staatsblad_event'
    source_pk           TEXT NOT NULL,                 -- composite PK in source table, serialised
    source_action_seq   INT NOT NULL DEFAULT 0,        -- r16: one source row can emit multiple actions (share_transfer=2, ma_event:merger=N+1). 0 for single-action sources; 0..N for multi-action.
    source_filing       TEXT,                           -- deposit_key for NBB-sourced edges
    source_rank         INT NOT NULL,                  -- 1=NBB shareholder, 2=NBB PI (reverse), 3=gazette transfer, 4=gazette capital, 5=other

    -- Time — bitemporal-ready but not bitemporal-required for v1
    fiscal_year         INT,
    deposit_date        DATE,                           -- when the source filing was deposited (transaction time)
    valid_from          DATE,                           -- when the ownership relation began (best estimate)
    valid_to            DATE,                           -- NULL = current

    -- Quality
    confidence          REAL DEFAULT 1.0,

    -- r16: include source_action_seq because one source row can produce
    -- multiple actions. share_transfer = 2 actions (close old, open new),
    -- ma_event:merger = N+1 actions. Without source_action_seq the second
    -- INSERT for the same source_pk would conflict on the old uniqueness rule.
    UNIQUE (source_table, source_pk, source_action_seq)
);

-- r17: split into TWO surfaces — current-state view + as-of function.
-- Earlier "ownership_edge_canonical" keyed by (parent, child, fiscal_year)
-- collapsed real valid-time intervals when there were multiple windows
-- within one year (e.g. shares transferred in March, transferred again in
-- August). The current-state question and the as-of-date question are
-- different queries; conflating them in one view is a footgun.

-- (1) Current-state view — only edges in force as of NOW.
-- Use this for the spiderweb / structure / network read paths that want
-- "who owns this company today".
-- r23 — also filter on valid_from. Same fix as administrator_current:
-- without it, future-dated edges (an ownership transfer scheduled to take
-- effect next month, recorded today) leak into "today's" walk.
-- r25 — accept NULL valid_from as "unknown start" per the convention
-- documented above administrator_current. Backfilled ownership data
-- from gazette events doesn't always have a structured start date.
CREATE OR REPLACE VIEW ownership_edge_current AS
  SELECT DISTINCT ON (parent_kind, parent_id, child_kind, child_id)
         id, parent_kind, parent_id, child_kind, child_id, fiscal_year,
         pct, edge_kind, source_rank, deposit_date,
         valid_from, valid_to
  FROM   ownership_edge
  WHERE  (valid_from IS NULL OR valid_from <= CURRENT_DATE)
    AND  (valid_to IS NULL OR valid_to > CURRENT_DATE)
  ORDER BY parent_kind, parent_id, child_kind, child_id,
           source_rank ASC, deposit_date DESC NULLS LAST;

-- (2) As-of-date function — edges in force on a specific date.
-- Use this for "ownership tree as of the previous deal" or any historical
-- diligence query. Returns multiple rows per (parent, child) only when
-- multiple sources describe the same edge (resolved by source_rank).
CREATE OR REPLACE FUNCTION ownership_edge_as_of(target_date DATE)
RETURNS TABLE (
    id BIGINT, parent_kind TEXT, parent_id TEXT,
    child_kind TEXT, child_id TEXT, fiscal_year INT,
    pct NUMERIC, edge_kind TEXT, source_rank INT,
    deposit_date DATE, valid_from DATE, valid_to DATE
)
LANGUAGE SQL STABLE AS $$
  SELECT DISTINCT ON (parent_kind, parent_id, child_kind, child_id)
         id, parent_kind, parent_id, child_kind, child_id, fiscal_year,
         pct, edge_kind, source_rank, deposit_date, valid_from, valid_to
  FROM   ownership_edge
  WHERE  valid_from <= target_date
    AND (valid_to IS NULL OR valid_to > target_date)
  ORDER BY parent_kind, parent_id, child_kind, child_id,
           source_rank ASC, deposit_date DESC NULLS LAST
$$;

-- semantics map to one of the above:
--   "current edge" → ownership_edge_current
--   "historical edge as of date X" → ownership_edge_as_of(X)
-- An executor finding a stale canonical reference should rewrite to one

-- UBO walk in pure SQL — no AGE needed.
-- r19: ubo walk uses ownership_edge_current for "today's" walk;
-- swap to ownership_edge_as_of(date) for historical walks.
-- r20: cycle-safe — path array tracks visited nodes so cross-ownership
-- (A→B→A or longer cycles, real or data-error) doesn't loop forever.
WITH RECURSIVE ubo(child_id, parent_id, depth, path) AS (
  SELECT child_id, parent_id, 1, ARRAY[child_id, parent_id]
  FROM ownership_edge_current
  WHERE child_id = '0400378485'
  UNION ALL
  SELECT u.child_id, e.parent_id, u.depth + 1, u.path || e.parent_id
  FROM ubo u
  JOIN ownership_edge_current e ON e.child_id = u.parent_id
  WHERE u.depth < 6
    AND NOT (e.parent_id = ANY(u.path))   -- cycle guard
)
SELECT * FROM ubo;

  2. Dedup pass: same edge declared in two sources (shareholder + participating_interest reverse) → keep one with the highest-trust source. Sum-to-100% sanity check per child per fiscal_year, log warnings.
  3. Dual-write: every NBB filing's store_governance_snapshot() now also writes to ownership_edge. Old tables remain.
  4. Replace existing queries: the spiderweb network view, the "parent companies" panel on the structure tab, the participating-interest list — switch to recursive-CTE queries against ownership_edge_current (for live state) or ownership_edge_as_of(target_date) (for historical walks). No CREATE EXTENSION age step in v1; AGE stays explicitly deferred until a feature demonstrably can't be expressed in SQL. The previous draft contradicted itself by deferring AGE in callouts but keeping it in the build steps; this version aligns.

Downsides

High

Apache AGE adds an operational dependency

AGE is a Postgres extension. It lags major Postgres releases — if Hetzner / a future managed-Postgres provider runs PG 17 or 18, AGE may not have a build yet. Stack Overflow answers are fewer; debugging a Cypher query is lonelier than debugging SQL.

Mitigation: the ownership_edge table is plain SQL — AGE is opt-in. If AGE breaks, fall back to recursive CTEs against the same table; no data loss. Pin AGE version explicitly; track AGE/Postgres compatibility before any Postgres upgrade.

High

UBO data is fundamentally incomplete

Belgium has a UBO register operated by FPS Finance, but it isn't open bulk data. Public access has been legally constrained / suspended (CJEU C-37/20 + C-601/20 ruling on the 5th AML directive in 2022 forced member-state registries to restrict open public access; Belgium's UBO portal switched to authenticated and controlled access only — see FPS Finance UBO register for the current policy, and the FPS Finance news feed for the access-restriction announcements). So even if we did get programmatic access, the licence wouldn't necessarily permit re-exposure on a third-party tool. Many chains terminate at "this company is owned by another company with no admin/shareholder data of its own" — the graph will honestly show the dead-end, but a customer expecting "show me the ultimate human owner of every Belgian company" will be disappointed by what's actually possible.

Mitigation: design the UI to be honest about data gaps ("chain ends at company X, no further data available — UBO register would be needed"); pursue licensed UBO access (authenticated, with redistribution rights — not just bulk download) or a Graydon / Companyweb data partnership as a parallel track. The licence question is load-bearing: a tool that re-exposes legally-restricted UBO data without authorisation is a regulatory-risk story, not a feature.

Medium

Two query languages confuse new contributors

Some queries in SQL, some in Cypher. For solo dev today: not a problem. For onboarding a contractor or a future hire: real friction.

Mitigation: document which question-types use which language ("graph traversal → Cypher; everything else → SQL"); wrap common Cypher patterns as Postgres functions so callers don't have to learn AGE syntax.

Medium

Cycles and pathological queries

Companies can cross-own each other (real or data error). OWNED_BY*1..N with no upper bound becomes an accidental DoS — Cypher evaluators recurse until something dies.

Mitigation: always cap depth (*1..6 not *); add a backend check that rejects unbounded traversal; cycle-detect at write time and flag for review.

Medium

Percentages don't sum to 100%

For a given child company, summing pct across edges might give 87%, 103%, or NULL. Sources double-count between shareholder filings and participating-interest declarations. Gazette events are point-in-time and overlap. "Show me ownership %" becomes "show me ownership % from which source, as of when, with what dedup".

Mitigation: a deterministic dedup function picks one source-of-truth per (parent, child, fiscal_year); expose all sources in a "details" panel; show "≈ 100%" rather than the literal sum where it's noisy.

Low

Storage growth from edge duplication

Same logical edge declared in shareholder + participating + gazette = three rows in ownership_edge. Acceptable at current scale (~50K shareholder rows + ~similar PI), but multiplies index footprint.

Mitigation: live with it; the tables remain small in absolute terms.

Low

Backup/restore needs verifying

AGE stores graph data in special schema tables. Some pg_dump options skip extension data. Easy to miss until restore-day.

Mitigation: add an end-to-end backup-restore test to the deploy checklist before going live with AGE.

Low

Graph visualisation is harder than it looks

Force-directed layouts of complex Belgian holding groups can look like a hairball. The current spiderweb view works because it's a small radius around a single CBE.

Mitigation: always visualise from one anchor node, with depth and edge-count caps; offer a tabular UBO walker as the primary view, graph as the "nice to have".

Open questions before this can start:
  • AGE or pure SQL? AGE is more elegant but adds dependency. Pure SQL with recursive CTEs is fine for the next 2 years; AGE is worth it if we genuinely need ad-hoc graph queries from analysts. Recommend: pure SQL first; layer AGE on later if real demand emerges.
  • Do we model person → company ownership too? Today personal shareholdings are sparse but growing.
  • What's the canonical source for percentage when multiple sources disagree? Suggest: most-recent NBB filing > gazette > participating-interest of declared parent.
3

Bitemporal — time as a first-class dimension

Today every relationship reflects "the latest filing's view". "What did this company look like in 2022?" is reconstructable but not modeled. Tomorrow every relationship has valid_from / valid_to / recorded_from/recorded_to; state_as_of(date) becomes a one-line query.

Cross-cutting requirements apply here — see the red table at the top of this page. Most-relevant items for the bitemporal layer: migration replayability (this is the largest and most-touchy schema change of the four — migrations runner with per-file tx mode + rollback pairing is non-negotiable), backup coverage (the new _fact data is the audit trail; verified-restore drills must include it), and per-request phase timing (the new _current views' cost profile differs from the base tables — must be measured, not assumed).
"Bitemporal" decoded: two independent time dimensions per fact. Valid time = when the fact is true in the real world (director appointed 2020-03-15, resigned 2023-06-30). Transaction time = when our DB learned about the fact (filing landed 2024-02-10). Most queries only need valid time; auditability needs both.

Schema sketch — v1 narrower than the original draft

The original draft proposed adding valid_from / valid_to / recorded_from / recorded_to to every relationship table across KBO, NBB, Staatsblad, and enrichment. That's a lot for one migration. Tightened v1: append-only fact tables for the relationship tables only, plus _current views. True bitemporality across KBO/Staatsblad/enrichment is a follow-up phase.

-- IMPORTANT REVISION: explicit "_current" views, NOT replacement.

-- and creating a current-only view AT the original name. That would silently
-- change behaviour for callers that intentionally read historical rows —

-- and the `include_historical` semantics in backend/routers/companies/network.py. Each call

-- (history-aware) or administrator_current (current-only).

-- Step 1a: add columns NULLABLE, NO DEFAULT (r16 fix).
-- ⚠️ Earlier draft had `recorded_at TIMESTAMPTZ DEFAULT NOW()` and then
-- backfilled `WHERE recorded_at IS NULL` — that doesn't work. The default
-- fills in immediately on row migration so the backfill condition matches
-- nothing. Add columns NULLABLE first; backfill; THEN set the default for
-- future inserts.
--
-- Also: bitemporal needs a transaction-time PAIR, not a single timestamp.
-- recorded_from = when this fact entered the DB; recorded_to = when it
-- was superseded by a newer fact (NULL = current). A query "what did the
-- DB say on date X about state at date Y" needs both bounds.
ALTER TABLE administrator
  ADD COLUMN valid_from           DATE,
  ADD COLUMN valid_to             DATE,
  ADD COLUMN source_deposit_date  DATE,         -- the NBB filing's deposit date; "as-of" anchor
  ADD COLUMN recorded_from        TIMESTAMPTZ,  -- when DB learned this fact
  ADD COLUMN recorded_to          TIMESTAMPTZ;  -- when superseded; NULL = current

-- nbb_load_log, etc. do NOT have a deposit_date column. deposit_date lives
-- on financial_data (src/schema.sql:130). Backfill JOINs through there.
-- DISTINCT because financial_data has many rows per (enterprise_number,

-- Step 3: create EXPLICIT _current view (not replacing the base table).
-- ⚠️ r21 — view MUST filter on BOTH valid_to AND recorded_to. r20 only
-- checked valid_to, which leaks superseded-but-still-valid facts into
-- "current" reads. A row with recorded_to set means a newer version of
-- the same fact exists; the old row should never appear in current.
-- r23 — also filter on valid_from. Without it, future-dated mandates
-- (valid_from > CURRENT_DATE — e.g. a director appointment whose mandate
-- starts next month) leak into "current" reads. Same fix needed on the
-- equivalent _current views for shareholder, participating_interest,
-- affiliation, and ownership_edge_current.
--
-- r25 — NULL valid_from convention: backfills will have rows where the
-- source filing didn't carry a usable start date (e.g. legacy KBO rows
-- from before NBB structured deposit_date was introduced). Two valid
-- treatments — pick per-table based on backfill quality:
--   (i)  TREAT NULL AS UNKNOWN-START (default for v1, lossy-tolerant):
--        the row is included in _current — convention is "we don't know
--        when this fact started being true, so we assume it has been
--        true at least since the filing was deposited". The IS NULL
--        branch below implements this. Document NULL valid_from as
--        "unknown start" in the column comment so callers know.
--   (ii) ENFORCE NOT NULL (post-backfill, once data quality verified):
--        ALTER COLUMN valid_from SET NOT NULL; drop the IS NULL branch.
--        Cleaner long-term, but only safe once we've measured backfill
--        completeness.
-- v1 starts on (i) for administrator + ownership_edge; tightens to (ii)
-- per-table once Stage's postcondition includes "SELECT count(*) FROM <t>
-- WHERE valid_from IS NULL = 0".
CREATE VIEW administrator_current AS
  SELECT * FROM administrator
  WHERE (valid_from IS NULL OR valid_from <= CURRENT_DATE)
    AND (valid_to IS NULL OR valid_to > CURRENT_DATE)
    AND recorded_to IS NULL;

-- Step 4: explicit fact view too, just as documentation of intent.
-- (The base table `administrator` IS the fact table; this view exists
-- only so call-sites that mean "history" can name it explicitly.)
CREATE VIEW administrator_fact AS
  SELECT * FROM administrator;

-- r20/r21 invariants — natural key + supersession + uniqueness enforcement.
-- Without these, superseded rows could keep recorded_to IS NULL forever
-- and the table would have multiple "current" rows per logical fact.
--
-- ⚠️ r21 corrections to the r20 natural-key sketch:
--   1. administrator column is `name`, NOT `person_name` (r20 typo).
--   2. Per-table keys must use search_normalize() + COALESCE on identifier
--      and country, not raw name only. Same-name shareholders + NULL
--      identifiers + foreign entities + renamed entities all break a
--      raw-name natural key.
--
-- ⚠️ r23 SUPERSEDES r22 on the role-nullability question. The r22 reading
-- ("role is nullable per src/schema.sql:322 column declaration") missed
-- src/schema.sql:327: PRIMARY KEY (enterprise_number, deposit_key, name, role).
-- Postgres makes every PK column implicitly NOT NULL, so on any DB created
-- from src/schema.sql, role IS NOT NULL — even without the explicit modifier
-- on line 322. The r22 COALESCE wrap is therefore unnecessary AND misleading
-- (implies role can be NULL when it can't). Reverted below.
-- The general principle stands for FUTURE columns added to natural keys:
-- if a column is genuinely nullable (not in a PK, not declared NOT NULL),
-- wrap it in COALESCE in both the natural-key listing AND the partial
-- unique index, OR use NULLS NOT DISTINCT (PG 15+), OR declare it NOT NULL.
-- Verify live DB nullability via `\d+ administrator` or pg_attribute.attnotnull
-- before relying on this for any new natural key.
--
-- Per-table NATURAL KEY (column-correct + identity-hash style):
--   administrator         → (enterprise_number, search_normalize(name), role)
--                           — `name` is the actual column; r20 said `person_name` in error
--   shareholder           → (enterprise_number, search_normalize(name),
--                            COALESCE(identifier, ''), COALESCE(country, ''))
--   participating_interest→ (enterprise_number, COALESCE(identifier, ''),
--                            search_normalize(name), COALESCE(country, ''))
--   affiliation           → (enterprise_number, search_normalize(person_name),
--                            via_enterprise_number)
--
-- For tables where the discriminator set still allows false supersession
-- (e.g. two genuinely different "Jan De Smet" shareholders with identical
-- normalised name + same enterprise + both with NULL identifier), the
-- safer path is an explicit identity-hash column populated at write time:
--   identity_hash = sha256_short(natural_key_components || '|' || coalesce(source_filing, ''))
-- This treats each filing's row as distinct unless explicitly deduped.
-- Pick per-table based on observed false-supersession risk in v1 data.
--
-- SUPERSESSION RULE: when a new fact with the same natural key arrives,
--   1. UPDATE prior row(s) with same natural key AND recorded_to IS NULL
--      SET recorded_to = NOW()
--   2. INSERT the new row with recorded_from = NOW(), recorded_to = NULL
-- Both steps in one transaction. store_governance_snapshot() implements this.
--
-- ENFORCEMENT — partial unique index for "at most one current row per
-- natural fact". This is a structural invariant; if it ever fails, the
-- supersession logic has a bug and we want the writer to error, not
-- silently corrupt the bitemporal model.
-- r23 — TWO corrections to the r22 sketch:
--   (i)  Reverted COALESCE(role, '') back to bare `role`. role is part of
--        src/schema.sql:327 PRIMARY KEY → effectively NOT NULL. r22 was wrong.
--   (ii) Tightened the partial filter to also require valid_to IS NULL.
--        The r22 filter `WHERE recorded_to IS NULL` over-constrained the
--        invariant: it would block legitimate "second mandate" rows for the
--        same (ent, name, role). Real example: Jan De Smet was director of
--        NV X 2018-2022, then director again 2024-present. Both rows have
--        recorded_to IS NULL (neither supersedes the other — they describe
--        non-overlapping valid-time intervals). The r22 index would reject
--        the second insert. Adding `AND valid_to IS NULL` narrows the
--        invariant to "at most one OPEN-ENDED CURRENT row per natural fact" —
--        which is what we actually mean. Past closed mandates with valid_to
--        set don't match the partial filter and are unconstrained.
--        Alternative considered: include valid_from as a key column. Equivalent
--        for this case but allows multiple OPEN-ENDED current rows differing
--        only by valid_from start, which is a state we never want.
CREATE UNIQUE INDEX idx_admin_current_natural
  ON administrator (enterprise_number, search_normalize(name), role)
  WHERE recorded_to IS NULL AND valid_to IS NULL;

-- fiscal_year, filing_json). Add deposit_date to the signature so the

-- ⚠️ r22 — INTERVAL SEMANTICS (half-open at storage; normalise at write).
-- Convention: valid_from <= D AND (valid_to IS NULL OR valid_to > D).
-- Same for recorded_from/_to. The as-of helpers below already match.
--
-- ETL CONTRACT for loaders writing valid_to from filing end dates:
-- filings state HUMAN-INCLUSIVE dates ("mandate ends 2026-12-31" means
-- served on the 31st). Add one day at write time:
--   valid_to = filing_end_date_inclusive + INTERVAL '1 day'
-- Docstring + column comment must say so. Storing raw = one-day off-by-one.
-- Exception: explicitly-exclusive source dates get stored raw.

-- "As-of" helper for time-travel queries. r10: takes BOTH valid_at
-- (when the fact was true) AND known_at (when the DB knew the fact).
-- Defaults to known_at=NOW() so callers that only care about valid time
-- can pass one argument; full bitemporal queries pass both.
--
-- Use cases:
--   admins_as_of('2024-01-01')                  → state at 2024-01-01 per current data
--   admins_as_of('2024-01-01', '2024-06-30')    → "what did our DB say on 2024-06-30 about state at 2024-01-01?"
-- The second form catches the "we filed in March, then a correction
-- arrived in May" pattern that valid-time-only queries can't see.
CREATE FUNCTION admins_as_of(
    valid_at DATE,
    known_at TIMESTAMPTZ DEFAULT NOW()
)
RETURNS SETOF administrator LANGUAGE SQL STABLE AS $$
  -- r17: filter on both transaction-time bounds (recorded_from / recorded_to),
  -- not the deleted recorded_at column. Without the recorded_to check, the
  -- helper returns superseded facts as if they were still authoritative on
  -- known_at — which defeats the purpose of bitemporal storage.
  SELECT * FROM administrator
  WHERE valid_from <= valid_at
    AND (valid_to IS NULL OR valid_to > valid_at)
    AND recorded_from <= known_at
    AND (recorded_to IS NULL OR recorded_to > known_at)
$$;

Downsides

High

Hardest to reverse — once you store history, you can't unstore it

Adding columns is easy. Removing them later loses information. The schema is reversible; the mental model of the application becomes hard to revert. Once the codebase expects "every query is implicitly current state", reverting to "rows are facts, no time" requires touching every query.

Mitigation: commit to it after Person and Ownership ship and you've validated the broader direction. Don't half-do this; either bitemporal or not.

High

Determining valid_from / valid_to is its own ETL problem

NBB filings tell you "as of fiscal year end". Gazette events give you the act date. KBO updates are "as of yesterday". Reconciling these is non-trivial:

  • A director appointed 2024-01-15 (gazette) doesn't appear in NBB FY2023 filing (closed 2023-12-31). Is the director "valid from 2024-01-15"? Probably yes.
  • FY2024 NBB filing (closed 2024-12-31) includes the director. Should we use the gazette's earlier valid_from, or the NBB's "as of 2024-12-31"? Use the earlier — gazette is closer to truth.
  • What if NBB and gazette disagree on the role? Honour the source with the most recent recorded_from (or, if both are still active, the higher-trust source per the source-rank table in §2).

Mitigation: spend week 1 of the build on the reconciliation rules — write them down, get them reviewed, add SQL test harnesses.

High

Migration scope on prod tables

administrator ~1.15M rows, shareholder ~43K, address ~3M, activity millions. ALTER TABLE ADD COLUMN with a non-volatile default is fast in modern Postgres, but backfilling values is not. Long lock-holding queries on prod = bad day.

Mitigation: ADD COLUMN with NULL default (instant); backfill in batches via a Python script with throttling; run during the existing nightly quiet window; keep a rollback migration prepared.

Medium

Storage growth — every fact lives forever

Today admin has 1.15M rows reflecting current state. With bitemporal we keep historical mandates too. Realistic growth: 30–50% over 2 years.

Mitigation: Postgres compression handles it; budget for disk; archive cold transaction-time rows after N years to cold storage if it ever bites.

Medium

Application code learns to ask "current or history"

Every query needs an "implicit current state" filter or it'll return historical rows alongside live ones. The _current views handle this for most cases, but new code must remember to use them, not the base tables.

Mitigation: keep the base table as-is (administrator stays the fact table by name, with new valid_* / recorded_from / recorded_to columns) and add an explicit administrator_current view. New code reads administrator_current by convention; history-aware code reads administrator directly. The read-path audit (a mandatory step) classifies every existing call site so nothing changes silently. 5b. End-to-end cancellation — stale queries stop grinding

Today: browser cancels via AbortController → backend keeps running query → DB connection held → next request waits. Tomorrow: cancellation propagates from browser → FastAPI → DB connection.

The bridge is not optional during the bridge period. Once the de-async ride-along ships (week 0 — sync handlers running in FastAPI's threadpool with psycopg2), pg_cancel_backend() from a separate connection is the only way to cancel a running query. Async task cancellation does not propagate into sync code running on a worker thread — psycopg2 has no awareness of FastAPI's cancellation signals. So during the entire bridge period (all of weeks 0 → asyncpg landing), the watchdog + dedicated cancel-pool pattern is the implementation, not "one option of several". The native asyncpg path replaces it only after §5a step 2 lands on the search hot path.

Schema / code sketch

# Bridge pattern (works on psycopg2; ship before asyncpg lands).
# A watchdog task per request polls Request.is_disconnected() and
# cancels the running query from a separate connection on disconnect.

async def _with_cancel_watchdog(req: Request, async handler):
    cancel_event = asyncio.Event()

    async def watch():
        while not cancel_event.is_set():
            if await req.is_disconnected():
                cancel_event.set()
                # Trigger DB-level cancel via a dedicated cancel connection
                cancel_running_query(req.state.query_pid)
                return
            await asyncio.sleep(0.05)

    watcher = asyncio.create_task(watch())
    try:
        return await handler()
    finally:
        cancel_event.set()
        watcher.cancel()

# Native pattern (asyncpg, post-migration).
# Task cancellation propagates to the connection automatically.
async def search_companies(...):
    try:
        return await _search_companies_async(...)
    except asyncio.CancelledError:
        # asyncpg has already issued the DB-level cancel
        raise

How it gets built

  1. Bridge ships first: ~40 lines of middleware. Adds the watchdog task to every search-route request. Polls Request.is_disconnected() every 50 ms.
  2. Tag the connection with a request token via application_name (r20/r21): when the request handler acquires its connection, it does SELECT set_config('application_name', 'datasnoop:rid=' || $1, false) with the request's UUID. application_name is one of the few connection-level settings that is visible across sessions via pg_stat_activity.application_name. Stash both captured_pid (from pg_backend_pid()) and captured_request_id on req.state for the watchdog. Earlier r17 draft used SET LOCAL ds.query_token + current_setting() — that was wrong; current_setting() evaluates in the calling session, not the target session, so it never sees the target's token. application_name is the right tool.

    r21 critical addendum — reset on return-to-pool: pooled connections inherit the previous request's application_name until reset. Without an explicit reset on every putconn, a later non-search route handler could acquire the connection still tagged datasnoop:rid=<old-uuid>, run a totally different query, and be cancelled by a stale watchdog from the previous request. Mandatory: db.py::put_connection() issues SELECT set_config('application_name', 'datasnoop', false) (no rid) before returning the connection to the pool. The token is only set during an active request.
  3. Cancel via dedicated connection — with PID + application_name ownership guard (r20): a separate small pool reserved for issuing cancels. Never reuse the connection that's running the query. The cancel SQL queries pg_stat_activity which IS cross-session-visible: 
    -- r20 ownership guard: cancel ONLY if the target connection STILL carries
-- our request's tag. application_name survives across the cancel session.
SELECT pg_cancel_backend(pid)
FROM   pg_stat_activity
WHERE  pid = :captured_pid
  AND  state = 'active'
  AND  application_name = 'datasnoop:rid=' || :captured_request_id;
    Without this guard, a watchdog whose request was cancelled at minute T+0 could fire pg_cancel_backend at minute T+0.5, hitting a connection that the pool has already reissued to a different request running its own (later) query. The application_name check makes that race impossible: if the pool reissued the connection, the new request's handler will have set its own application_name, and the cancel WHERE clause won't match.
  4. r25 — discard-or-rollback after QueryCanceled (mandatory; without it, a cancelled tx poisons the next request on that connection). pg_cancel_backend() aborts the running query but leaves the psycopg2 connection in aborted-transaction state. Returning that connection straight to the pool means the next caller's first statement errors with InFailedSqlTransaction on a query they didn't run — a classic poisoned-connection footgun. The handler's exception path MUST do one of:
    • Default — rollback then return: except psycopg2.errors.QueryCanceled: conn.rollback(); pool.putconn(conn); raise. Cheap; clears the abort state. Works for the search read path which doesn't hold server-side cursors or other residual state.
    • Heavier paths — discard the connection entirely: except psycopg2.errors.QueryCanceled: pool.discard(conn); raise. Costs one new connection on the next request but guarantees no residual state. Use this on any future cancellable path that opens server-side cursors, declares temp tables, or sets session-level state beyond application_name.
    Pool-side enforcement: backend/db.py::put_connection() checks conn.info.transaction_status before accepting the connection back — anything other than TRANSACTION_STATUS_IDLE triggers a discard regardless of what the caller intended. This is the safety net for handlers that forget to roll back; the explicit rollback above is the fast path.

    Reset application_name on the rollback path too: the existing r21 reset (SET application_name = 'datasnoop') runs as part of put_connection, so it covers both rollback-and-return and successful-completion. The discard path skips it (connection is dropped), which is fine — a fresh connection has no application_name set.
  5. Native asyncpg path: once asyncpg ships (5a step 2), the watchdog can be removed for the asyncpg-path queries — task cancellation handles it natively (and the token-guard layer becomes vestigial; remove it then). Note: asyncpg's task-cancellation path also leaves the connection in a usable state automatically — no analogous discard-or-rollback step needed once the bridge is gone.

5c. Full-text index on company names

Today's company-name search uses pg_trgm GIN exclusively. Trigram is great for typo tolerance ("colruty" → "Colruyt") but inefficient for tokenised matching of common words. A tsvector + GIN index does the latter natively. Postgres's FTS handles stop-words via the dictionary config and supports phrase-quoting + boolean operators via websearch_to_tsquery. r21 correction: earlier draft also claimed stemming, prefix matching, and BM25-style ranking — that was wrong for our setup. 'simple' dictionary doesn't stem (no language morphology applied), websearch_to_tsquery doesn't do automatic prefix matching (you'd need :* suffixes manually), and ts_rank_cd is a coverage-density score, not a true BM25 — it's good enough for "is this a hit" ranking but not a precision-tuned relevance model. See r18+r21 bench requirement: relative-to-trigram recall on every category, not absolute "BM25 quality".

Schema sketch

-- r16 SIMPLIFICATION: use an EXPRESSION INDEX, not a STORED generated column.
-- Earlier draft had ALTER TABLE ADD COLUMN ... GENERATED ALWAYS AS ... STORED
-- with a lock-caution comment because that triggers a table rewrite on 1.94M
-- rows. The simpler answer: skip the column entirely. Postgres can index an
-- expression directly, and the query side already calls the same expression.
-- No rewrite, no STORED storage cost, no synchronisation burden.

CREATE INDEX CONCURRENTLY idx_ci_name_tsv
  ON company_info USING GIN (to_tsvector('simple', name_normalized));

-- Same on denomination for trade-name match — also expression index:
CREATE INDEX CONCURRENTLY idx_denom_tsv
  ON denomination USING GIN (to_tsvector('simple', denomination_normalized));

-- ============================================================================
-- QUERY PATH: TWO APPLICATION-LEVEL QUERIES, NOT A SINGLE UNION ARM
-- ============================================================================
-- r10 simplification: don't UNION FTS with trigram in one SQL — run them
-- as two separate queries from the application. FTS first; trigram only on
-- underflow. Easier to instrument (per-arm latency) and reason about.
--
-- IMPORTANT: BOTH queries' :query parameter MUST go through the same
-- search_normalize() that built the indexed column. Otherwise "Colruyt NV"
-- query → tokens [colruyt, nv], but stored vector → tokens [colruyt] —
-- and the @@ never matches. Apply normalization in the application before
-- binding :query, OR call search_normalize(:raw_query) inline:

-- Query 1 — FTS primary. Both indexed-expression AND query go through
-- the same to_tsvector('simple', search_normalize(...)) — so the planner
-- uses idx_ci_name_tsv. The expression match is exact.
SELECT ci.enterprise_number, ci.name,
       ts_rank_cd(to_tsvector('simple', ci.name_normalized), q) AS fts_score
FROM   company_info ci,
       websearch_to_tsquery('simple', search_normalize(:raw_query)) q
WHERE  to_tsvector('simple', ci.name_normalized) @@ q
ORDER BY fts_score DESC
LIMIT 50;

-- ⚠️ r16 application-level guards before issuing Query 1:
--
-- A. Empty-normalization edge case: if search_normalize(raw) returns ''
--    (entire input was punctuation / legal-suffix-only), don't run FTS.
--    websearch_to_tsquery('') matches nothing AND wastes a planner cycle.
--    Fall back to trigram on the raw input or return empty result.
--
-- B. Leading-legal-suffix queries: search_normalize() strips ONLY trailing
--    legal forms (per docs/search.md "Trailing-anchor only — never kills
--    leading 'NV'"). So "NV Colruyt" normalises to "nv colruyt" — and
--    websearch_to_tsquery would require BOTH 'nv' AND 'colruyt' tokens to
--    match. The indexed value for "Colruyt NV" is just "colruyt" — no 'nv'.
--    Result: zero hits.
--    Mitigation: at the application layer, when the FIRST token of the
--    normalised query matches a known leading-suffix synonym (NV, BV, SA,
--    etc.), strip it and re-issue. Cheap, removes the asymmetry.

-- Query 2 — runs ONLY when Query 1 returns < 5 hits AND len(query) ≥ 4.
-- Pure trigram fallback for typos / partial matches.
SELECT ci.enterprise_number, ci.name,
       similarity(ci.name_normalized, search_normalize(:raw_query)) AS trgm_score
FROM   company_info ci
WHERE  ci.name_normalized %% search_normalize(:raw_query)
ORDER BY trgm_score DESC
LIMIT 50;

How it gets built

  1. Migration A (r17 — expression index, NO column added): CREATE INDEX CONCURRENTLY idx_ci_name_tsv ON company_info USING GIN (to_tsvector('simple', name_normalized)). Online (no table lock, no rewrite). Earlier draft proposed a STORED generated column with a lock-caution callout; the expression-index path avoids the rewrite entirely.
  2. Same for denomination: CREATE INDEX CONCURRENTLY idx_denom_tsv ON denomination USING GIN (to_tsvector('simple', denomination_normalized)).
  3. Bench against real queries — five required test categories before flipping primary (r18+r19.5): pull the top 100 unique queries from activity_log over the last 30 days AND deliberately augment with the categories below. Required pass criteria: recall ≥ trigram baseline on EVERY category before flipping; latency ≤ trigram on common-word queries.
    • (a) Trailing legal forms — "Colruyt NV", "Acme BV", "Caritas ASBL" — should match the trailing-suffix-stripped indexed value cleanly.
    • (b) Leading legal forms — "NV Colruyt", "BV Acme", "ASBL Caritas". search_normalize() strips trailing only (per docs/search.md: "Trailing-anchor only — never kills leading 'NV'"). So "NV Colruyt" → tokens [nv, colruyt], but the index for "Colruyt NV" → just [colruyt]. Without the application-layer leading-suffix-stripping guard (§5c implementation note), this category fails.
    • (c) Accent-stripping — "Müller" / "Muller" / "Mueller" should converge. Verifies unaccent + the normalisation chain holds.
    • (d) NL/FR multilingual variants — "Bruxelles" / "Brussel" / "Brussels"; "Pieter" / "Peter"; "François" / "Francois". The 'simple' tsvector config does NOT do stemming — these need to converge through normalisation, not stemming.
    • (e) Common Belgian surnames as queries — "Janssens", "De Smet", "Peeters". High homonym density; verifies the FTS path doesn't blow up on common-token queries that match thousands of entities.
    What FTS does NOT give you (r18 reminder): 'simple' + websearch_to_tsquery is not BM25, not stemming, not automatic prefix matching. It's a token-set match with phrase-quoting and basic boolean. Don't promise relevance behaviour the index doesn't deliver.
  4. Wire the FTS query as a separate application-level query, NOT a UNION arm (matches the §5c "two app-level queries, not single UNION" decision near line 3400 — r23 cleanup of an earlier draft contradiction). Sequence: app issues the FTS query first; if it returns < 5 hits AND length(query) ≥ 4, app then issues the trigram query as typo-fallback. Two queries are simpler to instrument (per-arm latency + hit count surface in Server-Timing headers cleanly) and easier to roll back arm-by-arm than a single SQL with an FTS arm bolted into _SEARCH_SQL.
  5. Smoke-test on staging against the existing scripts/smoke_search_v2.sh harness.
  6. Ship to prod with the activity_log click-rate as the regression signal — if click-rate drops > 5% on the new path, roll back.

    • r24 — migration style contract (enforced by the runner; lint-checked by CI). Every migration in migrations/ conforms to a small set of defaults so the executor doesn't have to re-derive safe-DDL practice each time. Runner sets these as session defaults at the top of every applied file; per-migration overrides via header comments:
      • SET lock_timeout = '5s'. A migration that can't acquire a lock in 5s aborts loudly rather than queuing behind a long-running query and starving readers. Override via -- @migration: lock_timeout=30s for migrations on tables with known lock contention (KBO daily-update table during the 06:00 UTC cron, etc.). NEVER override above 60s without operator approval — past that, fix the contention root-cause.
      • SET statement_timeout = '60s' as the default for tx-mode migrations. no-tx migrations override per-statement (e.g. CREATE INDEX CONCURRENTLY on company_info = ~25 min, set SET statement_timeout = '45min' at the top of that one file).
      • Large-table backfills run as batched UPDATEs in a loop, NOT as a bare UPDATE company_enrichment SET .... Pattern: DO $$ DECLARE batch_id BIGINT := 0; BEGIN LOOP UPDATE ... WHERE id BETWEEN batch_id AND batch_id + 10000 RETURNING ... ; EXIT WHEN NOT FOUND; batch_id := batch_id + 10000; PERFORM pg_sleep(0.1); END LOOP; END $$;. Batch size 10k for tables < 1M rows; 50k for 1–10M; the 100ms sleep keeps replication / vacuum from falling behind. A bare UPDATE on a 1.94M-row table holds row-locks until commit, blocks every concurrent writer, and bloats the table.
      • Constraints add online via NOT VALID + VALIDATE CONSTRAINT. ALTER TABLE foo ADD CONSTRAINT foo_check CHECK (...) NOT VALID; in one migration; ALTER TABLE foo VALIDATE CONSTRAINT foo_check; in a follow-up. The first takes a brief schema-modification lock; the second scans the table without blocking writes. Bare ADD CONSTRAINT takes an ACCESS EXCLUSIVE lock for the duration of the table scan and is forbidden on tables > 100k rows.
      • Hot-path indexes ship with EXPLAIN evidence in the PR description. Required: EXPLAIN (ANALYZE, BUFFERS) ... output for the target query against staging data BEFORE and AFTER the new index, plus a one-sentence "this is the path the optimiser picked" interpretation. Without the AFTER EXPLAIN, the index might not be used — Postgres planner picks based on stats, not the developer's intent. The replay CI gate doesn't catch unused indexes; this evidence does. Hot-path = anything called from a route the user waits on (search, profile, screener); non-hot-path indexes (admin, batch jobs) need only the AFTER EXPLAIN.
